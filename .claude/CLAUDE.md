# General Rules
- 常に日本語で返答する。すべての応答、説明、コメントは日本語で行う。
- コードを参照、変更、追加する場合はまずdocsディレクトリにある仕様書類を参照
- コードに対して変更、追加を行った場合はdocsディレクトリにある仕様書類も合わせて変更し一貫性を保つ
- 仕様書を作成する場合は、機能ごとにマイルストーンをM{番号}で採番
- 実装計画を立てる際はマイルストーンに紐づけてチケットをT-{番号}で採番
- /batchコマンドで複数エージェントによる実装を行うときはgit worktreeでお互いの変更が干渉しないようにする。サブエージェントのタスク終了後は使用していたworktreeを削除する。 

# Git Rules
- claude codeによる変更をコミットする場合はuser.name=claude user.email=noreply@claude.comでコミットを作成する
- ユーザーからの指示が完了したらgit statusを実行し、変更があればコミットを作成する
- コミット後にgit pushは行わない。pushはユーザーが手動で行う
- 新しく機能を実装する場合はブランチを切ってコミットする
- ブランチはマイルストーンごとに作成する
  - 例：`feature/m1-add-feature`
- コミットはチケットが完了するごとに行う
- コミットメッセージは `M{番号}: T-{番号} {日本語の要約}` の形式で行う
  - 例： `M1: T-005 SearchRouterの骨組みを作成`
- マイルストーンの全チケットの実装、テストが完了したら--no-ffオプションでmainブランチにマージする
- マージ完了後のブランチ削除はユーザーの確認を得てから行う

# Tool Usage Rules
- ホスト側でPythonを実行する際は必ず `uv` コマンドを介して起動し、直接 `python` 実行を避ける（例: `uv run python script.py`）
- 技術スタックを使用する場合はdocker composeで管理する
- webUIに関する問題の確認と変更後は、playwrightを使用してスクリーンショットを/tmpに保存して表示崩れがないか確認する
